## Section 01 소프트웨어의 이해

### 1. 소프트웨어가 사용되는 곳

소프트웨어는 금융, 의료, 항공기, 자동차, 건물, 스마트폰, 노트북 등 다양한 분야에서 서비스 형태로 활용된다.

---

### 2. 소프트웨어의 정의

- **프로그램(좁은 의미)**: 원시코드(source code)만을 의미
- **소프트웨어(넓은 의미)**: 프로그램 그 이상의 것을 포함하는 포괄적 개념
  - 원시코드(source code)
  - 모든 산출물 (자료구조, DB구조, 테스트 결과 등)
  - 각 단계마다 생산되는 문서
  - 사용자 매뉴얼

---

### 3. 소프트웨어의 특징

#### 제조가 아닌 개발
| 구분 | 특징 |
|------|------|
| 제조 (H/W) | 정해진 틀에 맞춰 일정하게 생산, 많은 인력 필요, 능력별 결과물 차이가 근소함 |
| 개발 (S/W) | 개인 능력별 결과물 차이가 매우 큼 |

#### 소모가 아닌 품질 저하
| 구분 | 특징 |
|------|------|
| H/W | 오래 사용하면 부품이 닳고, 고장 발생 빈도 높고, 기능도 떨어짐 |
| S/W | 오래 사용해도 닳지 않고, 고장 발생 빈도 낮고, 기능도 동일함 |

---

### 4. 실패 곡선 비교

#### (1) H/W 실패 곡선 — 욕조 곡선(Bathtub Curve)

```
실패율
  ↑
높음│▓▓▓
   │   ▓▓▓
   │      ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓       ▓▓▓▓
   │                               ▓▓▓▓▓▓
낮음│
   └──────────────────────────────────────────→ 시간
       [초기 실패]      [안정기]       [마모기]
```

| 구간 | 실패율 | 원인 |
|------|--------|------|
| 초기 | 높음 | 제조 불량, 초기 결함 |
| 안정기 | 낮음 | 오류 해결 후 안정 운용 |
| 마모기 | 다시 증가 | 먼지, 진동 등 주변 환경 문제 |

**흐름**: 초기 실패율 높음 → 오류 해결 → 오랜 기간 안정 사용 → 주변 환경 문제 발생 → 다시 실패율 증가

---

#### (2) 이상적인 S/W 실패 곡선

```
실패율
  ↑
높음│▓▓▓
   │   ▓▓▓
   │      ▓▓▓
   │         ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓ ← 폐기 시까지 낮게 유지
낮음│
   └──────────────────────────────────────────→ 시간
       [초기]              [안정기 — 이상적 조건]
```

- **조건**: 개발 완료 후 변경 사항 없어야 하고, 환경 변화 없어야 함
- **특징**: 발견되지 않은 오류로 초기 실패율은 높지만, 오류 해결 후 폐기 시점까지 낮은 실패율 유지

**흐름**: 초기 실패율 높음 → 오류 해결 → **폐기될 때까지 같은 낮은 실패율 유지**

---

#### (3) 실제 S/W 실패 곡선

```
실패율
  ↑                ↑변경        ↑변경       ↑변경
높음│▓▓▓            ▓▓▓▓        ▓▓▓▓▓      ▓▓▓▓▓▓
   │   ▓▓▓        ▓    ▓▓     ▓     ▓▓▓  ▓▓      ▓▓▓ ← 실제 곡선
   │      ▓▓▓▓▓▓▓▓       ▓▓▓▓▓         ▓▓
   │  _ _ _ _ _ _ _ _ _ _ _ _ _ _ _ _ _ _ _ _ _  ← 이상적 곡선
낮음│
   └──────────────────────────────────────────────→ 시간
```

| 이벤트 | 영향 |
|--------|------|
| 기능 추가 또는 수정(변경) 발생 | 부작용(Side Effect) 발생 → 실패율 급증 |
| 변경 반복 | 이상적 곡선과 실제 곡선의 격차가 점점 벌어짐 |

**흐름**: 초기 실패율 높음 → 오류 해결 → 실패율 낮음 → **변경 발생** → 부작용 → 실패율 급증 → (반복)

---

#### 세 곡선 비교 요약

| 구분 | 초기 | 안정기 | 장기 |
|------|------|--------|------|
| H/W (욕조 곡선) | 실패율 높음 | 낮음 | 다시 높아짐 (마모) |
| 이상적 S/W | 실패율 높음 | 낮음 | 계속 낮음 (폐기까지) |
| 실제 S/W | 실패율 높음 | 낮음 | 변경마다 급등 반복 |

> **결론**: 실제 소프트웨어는 변경이 발생할 때마다 실패율이 급격히 증가하며,
> 이것이 소프트웨어 품질 관리와 공학적 접근이 필요한 핵심 이유이다.

---

### 5. 소프트웨어 개발의 어려움

#### 발전 속도 문제
- H/W 발전은 빠르지만(PC·스마트폰의 크기, 속도, 성능), S/W 발전 속도는 상대적으로 느림
- H/W는 검증된 부품을 조립하는 형태인 반면, S/W는 처음부터 만들어가는 개발 형태
- **해결 방안**: CBD(Component Based Development) 개발 방법론

#### 관리의 필요성
- S/W 개발에도 비용 관리, 일정 관리, 개발자 관리 등 **도구를 활용한 적극적인 프로젝트 관리** 필요

#### 규모에 따른 어려움
| 규모 | 비유 | 특징 |
|------|------|------|
| 소규모 | 개집 짓기 | 설계 도면 불필요, 혼자 가능, 단순한 과정 |
| 중규모 | 단독주택 짓기 | 설계 도면·건축 설계사 필요, 많은 인원 참여 |
| 대규모 | 대형 빌딩 짓기 | 대형 장비·내진 설계·통제·조정 조직 필요, 고려 사항 多 |

#### 대규모 소프트웨어 개발의 3대 어려움
1. **개발 과정이 복잡하다** → 복잡함을 줄이기 위한 방법과 기술 제시
2. **참여 인력이 많다** → 의사소통 경로 증가, 협력 어려움 → 팀 구성 및 관리의 효율적인 방법 제시
3. **개발 기간이 길다** → 진행 상황 파악 어렵고 비용 산정도 어려움 → PMBOK 활용

## Section 02 공학과 소프트웨어 공학의 이해

### 1. 공학이란?

- **특성**: 과학적 지식을 활용하여 문제를 해결하되, **정해진 기간과 비용의 제약**을 받음

### 2. 소프트웨어 공학의 정의

> 품질 좋은 소프트웨어를 경제적으로 개발하기 위해 계획을 세우고, 개발하며,
> 유지 및 관리하는 전 과정에서 공학, 과학 및 수학적 원리와 방법을 적용하여
> 필요한 이론과 기술 및 도구들에 관해 연구하는 학문

- **취지**: 소프트웨어 개발 과정에 공학적인 원리를 적용
- **목적**: S/W 개발의 어려움 해결 / 생산성 향상 / 고품질 소프트웨어 생산

### 3. 소프트웨어 개발 생명주기 (SDLC)

```
계획 → 분석 → 설계 → 구현 → 테스트 → 유지보수
```

## Section 03 소프트웨어 개발 단계의 소개

### 1단계: 계획

- **개발 비용 산정**: COCOMO 모델, 기능점수(FP) 모델
- **일정 계획**: WBS(작업분할구조도), CPM(Critical Path Method)
- **위험 관리**

> **WBS**: 프로젝트 목표 달성을 위해 필요한 전체 업무를 최하위 작업 패키지 단위로 계층화하여 세분화한 구조도

### 2단계: 요구분석

| 방법론 | 사용 도구 |
|--------|-----------|
| 구조적 방법론 | DFD, DD, Mini Spec |
| 정보공학 방법론 | E-R 다이어그램 |
| 객체지향 방법론 | UML 유스케이스 다이어그램 |

- **최종 산출물**: 요구 분석 명세서

### 3단계: 설계

- **설계 원리**: 분할과 정복, 추상화, 단계적 분해, 모듈화, 정보은닉
- GoF의 디자인 패턴
- **모듈 평가 기준**: 응집도(Cohesion)와 결합도(Coupling)

### 4단계: 구현

- 프로그래밍 언어의 역사 / 표준 코딩 규칙 준수

### 5단계: 테스트

- 테스트 절차, 개발자/사용자 시각 분류, 목적별 분류, 품질 특성별 분류, 개발 단계별 분류

### 6단계: 유지보수

| 종류 | 설명 |
|------|------|
| 수정 유지보수 | 오류 수정 |
| 적응 유지보수 | 환경 변화에 대응 |
| 기능보강 유지보수 | 새로운 기능 추가 |
| 예방 유지보수 | 미래 문제 예방 |
